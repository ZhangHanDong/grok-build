# 写作风格约定（STYLE.md）

这本书的技术材料是扎实的。它此前的问题不在内容，在文风：过度整齐、过度强调、
过度升华——18 章长得像 18 份同模板的技术报告，缺少一本书该有的轻重与节奏。

消除"AI 味"的方向不是更口语，而是更克制。下面的规则按"能不能机械检出"分成两档：
硬规则（`cite_heal` / 风格软警告可查）与手感规则（靠冷读把关）。

## 一、硬规则

1. **一层开场 + 一层收束**。每章开头一个简短"定位"框；结尾只留一处收束——
   "模式提炼"与"设计要点回顾"**二选一**，不要两处都把正文再总结一遍。
   版本基准集中在前言，章末"版本演化说明"只记录本章特有的例外。

2. **每段至多一个粗体**。粗体是"这里请慢一点"的路标；一段里三四个粗体等于没有
   路标。整章粗体是稀缺资源，不是强调习惯。

3. **禁止自我认证**。删掉"诚实/如实/坦白/值得/尤其见功力/最漂亮的一手/教科书式"
   这类作者对自己或对代码的表扬。可信度由边界条件和证据产生，不由作者声明产生。
   反例：`验证流程有三处设计尤其见功力……其三，也是最漂亮的一手`。
   正例：`验签只证明配置曾由可信密钥签发，不证明磁盘上的 TOML 未被替换；
   check_on_disk_matches 因而再做一次逐字节比较`。

4. **比较只留差异轴**。禁止"其一/其二/其三"堆叠列举，禁止"真正的差异在于"这类
   自我加冕。把两个方案压到它们分岔的那一层，用一句话说清各自的代价。

5. **断言必带边界**。任何"因为 X 所以 Y"都要说清它在什么前提下成立、何时失效。
   把绝对句（"永不竞争""不改变任何决策"）改成带条件的句子。

## 二、手感规则（冷读把关）

6. **用现场进入，不用术语铺垫**。开头给一个具体场景或一次真实的调查，让读者先
   看见问题，再看代码，最后才抽象。别先摆一组名词。

7. **作者声音放在发现过程里**。第一人称只用于真实的调查路径，例如
   "注释写着 mmap，依赖表里却没有 memmap，翻到实现是 `fs::read`"。不写
   "本书如实呈现"。

8. **每章挑一种叙事形态**，别都套同一个模板：
   - 生命周期：沿一次 prompt / 一次流式输出推进（ch3–5、ch13–16）
   - 事故/威胁驱动：从一次 SIGBUS、一条注入攻击进入（ch6、ch11）
   - 方案对照：同一任务两种做法并置（ch9、ch12）

9. **引用减密度**。不是每句都挂 `file:line`。一段里把支撑性引用聚到句末或段末，
   只有最承重的定义/证据才值得单独内联。行号漂移由 `book/tools/cite_heal.py`
   机械维护，作者不必为"锚得准"而堆引用。

## 三、禁用/慎用词

| 慎用 | 换成 |
|---|---|
| 值得、值得一提、值得注意 | 直接说那件事，删掉"值得" |
| 真正的、真正、本质上 | 删，或换成具体的名词 |
| 其一、其二、其三 | 差异轴 / 直接分句 |
| 诚实、如实、坦白 | 删——用边界条件代替声明 |
| 尤其、最、极致 | 删形容词，留机制 |
| 精妙、漂亮、优雅、教科书式 | 删——让机制自己说话 |

## 四、流程配套

- `project.spec.md` 不再强制"章章 codex 对照"与"最低篇幅"。codex 对照只在真有
  差异轴可讲时出现；篇幅由内容决定，不由下限驱动。
- `check_chapter.py` 的字数下限、引用数下限降为**软警告**，不作硬门禁——门禁只
  保留"引用不越界"这类客观项，风格由冷读定。
- 每章改完请找两名目标读者冷读，只问三件事：哪里开始跳读、哪里需要回读、
  读完能否复述主线。
